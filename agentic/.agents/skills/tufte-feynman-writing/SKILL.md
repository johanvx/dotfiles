# Tufte/Feynman Writing (General)

Use this skill for blog posts, reflective essays, technical notes, and concise explainers.

## Core objective

Turn unstructured context into:
1. one clear takeaway,
2. a tight structure,
3. concise final prose,
4. a transparent record of what was cut and why.

---

## Mandatory intake questions (ask first)

1. Audience
2. Target length (e.g., 1–3 min read)
3. One takeaway sentence (draft allowed)
4. Writer perspective/constraints
5. Raw notes/context
6. Preferred output mode:
   - outline only
   - guided writing prompts
   - full draft
   - full draft + revision notes
7. Tone preference (default: clear, grounded, conservative)

If inputs are incomplete, ask up to 5 targeted follow-ups, then proceed with explicit assumptions.

---

## Standard workflow

### Step 0 — Preflight
Always output first:
1. Assumptions
2. Missing info
3. Risks (ambiguity, weak evidence, scope creep)
4. Plan

### Step 1 — Context triage (“too many context” support)
Build an item table:

| Item | Claim | Evidence/example | Relevance to user constraint | Keep/Cut |
|---|---|---|---|---|

Scoring guidance:
- +2 directly actionable under constraints
- +1 useful background
- 0 interesting but nonessential
- -1 off-topic

Keep top 3–5 items.

### Step 2 — Convert to cards
For each kept item:
- Claim
- Evidence
- Implication for this audience

### Step 3 — Structure before prose
Use one structure from `templates/post-structures.md`.

### Step 4 — Draft
Rules:
- one sentence, one idea
- mechanism/evidence > adjectives
- explicit transitions where needed
- no inflated claims

### Step 5 — Compression + flow pass
- remove repetition/filler
- improve transitions
- tighten verbs and nouns

### Step 6 — Main output assembly
Prepare the requested writing deliverable (outline/prompts/draft/revision).

---

## Standard output contract (every run)

1. Assumptions
2. Missing info
3. Risks
4. Main output(s)
   - One-line takeaway
   - Outline
   - Final draft or revision guidance (as requested)
5. Evidence/citation/integrity flags
   - `[EVIDENCE NEEDED: <claim>]`
   - `[REFERENCE NEEDED: <claim>]` (if source support is expected)
   - `[VERIFY DETAIL: <statement>]`
6. Next actions for user
   - What was cut (with reason)
   - Open gaps / suggested additions

---

## Style constraints

- Clarity over ornament.
- Evidence over hype.
- Narrow scope over comprehensive summary.
- Show limits/uncertainty where relevant.
- End with concrete next step or decision.

---

## Supported modes

### A) Coaching mode
User writes; agent gives prompts, structure, and edits.

### B) Co-writing mode
Agent drafts from notes, then explains reasoning and cuts.

### C) Rewrite mode
Agent revises user draft for higher signal-to-noise and better flow.

---

## Safety / integrity policy

- Do not invent facts.
- If a statement lacks support, mark:
  - `[EVIDENCE NEEDED: <claim>]`
- If source support is missing, mark:
  - `[REFERENCE NEEDED: <claim>]`
- If timeline/sources/details are uncertain, mark:
  - `[VERIFY DETAIL: <statement>]`
