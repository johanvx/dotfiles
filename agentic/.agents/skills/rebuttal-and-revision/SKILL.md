# Rebuttal and Revision (CS + Engineering)

Use this skill when the user needs help responding to reviewer comments, planning revisions, or rewriting sections after review.

## Non-negotiable safety policy

1. NEVER fabricate citations, results, experiments, or implementation details.
2. If evidence is missing, mark: `[EVIDENCE NEEDED: <claim>]`.
3. If references are missing, mark: `[REFERENCE NEEDED: <claim>]` + keywords.
4. Never use hostile or defensive language toward reviewers.
5. Always separate:
   - what was changed,
   - what was clarified,
   - what remains out of scope (with justification).

---

## Mandatory intake (ask before drafting)

1. Venue and decision status (major/minor revision, reject-resubmit, etc.)
2. Full reviewer comments (R1/R2/AE/Editor)
3. Current manuscript version (or relevant sections)
4. What was already changed
5. What cannot be changed (time/data/resource constraints)
6. New results available? (yes/no + summary)
7. Citation updates available? (yes/no)
8. Desired output:
   - response letter
   - per-comment response matrix
   - revised section text
   - all of the above

If missing key data, ask up to 5 targeted follow-ups, then continue with placeholders.

---

## Standard workflow

### Step 0 — Preflight
Output:
1. Assumptions
2. Missing info
3. Risks
4. Proposed response strategy

### Step 1 — Parse comments
Classify each comment as:
- Technical correctness
- Novelty/significance
- Experimental sufficiency
- Clarity/organization
- Citation/background
- Scope mismatch

### Step 2 — Build response matrix
For each comment, produce:
- Summary of concern
- Planned action
- Evidence required
- Manuscript location to revise
- Draft response text (polite, specific)

### Step 3 — Draft outputs
Depending on user request:
- response letter
- point-by-point responses
- revised manuscript paragraphs

### Step 4 — Calibration pass
Ensure:
- conservative claims
- no overpromising
- explicit limitations where unresolved

### Step 5 — Main output assembly
Prepare the requested deliverables:
- response letter draft
- response matrix
- revision checklist
- revised manuscript paragraphs (if requested)

---

## Standard output contract (every run)

1. Assumptions
2. Missing info
3. Risks
4. Main output(s)
   - response letter draft
   - response matrix
   - revision checklist
   - revised text (if requested)
5. Evidence/citation/integrity flags
   - `[EVIDENCE NEEDED: <claim>]`
   - `[REFERENCE NEEDED: <claim>]`
   - `[VERIFY DETAIL: <statement>]`
6. Next actions for user
   - unresolved technical gaps
   - unresolved reference gaps
   - prioritized revision order

---

## Tone requirements

- Respectful, concise, non-defensive
- Thank reviewer briefly, then address specifics
- Prefer factual language over rhetorical emphasis
- If disagreeing, do so with evidence and boundaries

Suggested phrases:
- “We thank the reviewer for this important point.”
- “We have clarified this in Section X, Paragraph Y.”
- “We agree and have added …”
- “We respectfully note that … under the scope of this study …”
